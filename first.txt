再来说说我对 Skills 的理解。我认为它本质上是从 Prompt 演变而来的：

初级阶段： 我们需要写大量标准且准确的长 Prompt，效率极低。

中级阶段： 出现了系统提示词（System Prompt），像是一个预设的“人设面具”，但它过于死板，缺乏灵活性。

高级阶段： 也就是现在的 Skills。

很多人会担心：如果我给 Agent 装了 50 个 Skills，它岂不是要一次性读完？那 Token 消耗岂不是要爆炸？AI 的注意力会不会发散到“既想干这，又想干那”？

我的回答是：No. 这种情况其实是“系统提示词”时代的弊端。Skills 引入了一个极其核心的机制——渐进式披露（Progressive Disclosure）。说人话就是：按需加载，用多少拿多少。 只有当 AI 判断当前任务需要某项技能时，相关的指令才会进入上下文。这既保住了 Token，也保住了 AI 的专注力。
至于 MCP (Model Context Protocol)，你可以把它想象成 AI 界的 “USB 接口标准”。
在 MCP 出现之前，如果你想让 AI 连接到你的 GitHub、Google Drive 或本地数据库，每个软件都需要编写一套专属的“翻译代码”；有了 MCP，只要插件符合这个标准，AI 就可以直接“盲插”并无缝调用这些外部数据。
看到这你会觉得 MCP 和 Skills 很像。确实，如果说 Skills 是赋予 AI 的“认知补丁”（教它怎么想），那么 MCP 就是 AI 的“物理接口”（让它能碰到数据）。它们一个优化了大脑的效率，一个打通了四肢的连接，共同把 AI 从一个“聊天框”变成了一个真正的“行动派”。

Workspace 工作区
是什么
Agent Manager 支持同时管理多个工作区，每个工作区对应一个文件夹。
如何打开新工作区
- 点击左侧边栏中的按钮
- 选择一个起始文件夹作为工作区根目录
如何切换工作区
- 通过左侧边栏在不同工作区的对话之间随意切换（最上面偏右侧有open Agent Manager）
核心概念：工作区 = 文件夹，一个工作区可以有多个对话，多个工作区可以同时运行、自由切换。 简单来说我们可以对同一个项目文件开两个窗口同时处理 （对于这个功能大家可以看B站 是很有用的功能）

Rules（规则）是手动定义的约束条件，引导 Agent 遵循特定行为、风格和技术栈偏好。本质上是一个 Markdown 文件，字符限制 12,000 字符。
存放位置
类型	路径
全局	~/.gemini/GEMINI.md
工作区	.agent/rules/（工作区或 git 根目录）
如何创建
... 下拉 → Customizations → Rules → + Global 或 + Workspace
激活方式（4种）
方式	说明
Manual	在输入框中 @提及 手动激活（推荐）
Always On	永远生效（推荐）
Model Decision	AI 根据自然语言描述自行判断是否应用
Glob	匹配文件模式时自动应用（如 *.js、src/**/*.ts）
Workflows（工作流）是定义一系列步骤，引导 Agent 完成重复性任务（如部署服务、处理 PR 评论）。也是 Markdown 文件，字符限制 12,000 字符，包含标题、描述和步骤。
存放位置
与 Rules 相同，分全局和工作区两级。
如何创建
... 下拉 → Customizations → Workflows → + Global 或 + Workspace
如何执行
在 Agent 输入框中使用斜杠命令：
/workflow-name
高级特性
- Workflow 嵌套调用：一个 Workflow 内可以调用其他 Workflow
    /workflow-1 → 调用 /workflow-2 → 调用 /workflow-3
  - AI 自动生成 Workflow：手动操作完一系列步骤后，直接让 Agent 根据对话历史生成 Workflow
对于这两个rules和workflows我希望大家可以和ai商量着去写 只要你的prompt足够清晰（一开始不够清晰也可以 和ai对话几轮 .md迭代几轮之后也是可以的）ai给你的效率提升是很大的 还有一点不要和ai比脑子 你有的只有你的判断力
Agent 模式与设置总结
一、对话级别模式（每次新对话可选）
Planning	深度研究、复杂任务、协作工作
Fast	简单任务（改变量名、跑几条命令等）
二、全局设置（Settings → Agent 标签页）
1. Artifact Review Policy（产物审查策略）
Request Review	Agent 规划完成后停下来等你审查，你可以添加评论修改方案
Always Proceed	Agent 规划完直接继续执行，无需手动审查
> 不需要手动审查计划 → 选 Always Proceed
> 想在 AI 动手前检查方案 → 选 Request Review
---
Terminal Command Auto Execution（终端命令自动执行）
选项
Request Review
Always Proceed
- Allow List 和 Deny List 均可在设置中自定义配置
举例：
- 设置为 `Request Review` + Allow List 加入 `git status`
  → 其他命令都要审查，但 `git status` 直接执行
- 设置为 `Always Proceed` + Deny List 加入 `rm -rf`
  → 其他命令都自动执行，但 `rm -rf` 需要你手动审查确认
（我觉得没什么用 主要是看看你是拿来干什么了 你如果想24小时让他工作 你自己还不想管 你就Always Proceed）
- Unix Shell → 按空格分词，匹配命令的前缀
- PowerShell → 匹配命令 token 的任意连续子序列
{Unix Shell — 前缀匹配
按空格把命令拆成 token，你配置的内容必须是命令开头的部分。
配置：git
匹配：git commit、git push、git status  ✅
不匹配：npm run git  ❌（git 不在开头）
配置：git push
匹配：git push origin main  ✅
不匹配：git commit  ❌（开头不是 git push）
---
PowerShell — 任意连续子序列匹配
token 可以匹配命令的任意位置，不必在开头。
配置：push origin
匹配：git push origin main  ✅（中间包含 push origin）
匹配：git push origin  ✅
不匹配：git origin push  ❌（push 和 origin 不连续）}
Agent Non-Workspace File Access（工作区外文件访问）
- 默认情况下 Agent 只能访问：
  - 当前工作区文件

最后讲一下关于Source Control：
他在Antigravity的最左侧的一栏中（logo像Y）
会出现两个选项第一个initializa repository和publish to GitHub
我先说一下我们为什么要用source control 因为你的项目需要迭代好多次 比如说我们点击initializa repository之后便可以进行commit 每次提交之后都会有记录 我们可以随时查看我们上次的更改 而且每次更改的时候都会让你写一个COMMIT_EDITMSG 你在第一行写你为什么更改或者这次更改的目的是什么（写给自己或者别人看的 让别人更好的理解这个项目）然后修好之后点击√之后就可以提交了 （如果有GitHub仓库 连接Github MCP也可以 但是这个是要在docker运行的情况下运行）
